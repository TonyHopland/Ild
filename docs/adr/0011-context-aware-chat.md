# ADR-0011: Context-aware chat — fixed scratch cwd, worktree by add-dir, client-applied loop edits

The per-user chat ([ADR-0010](./0010-standalone-chat-session.md)) is made aware of what the user has open in the UI via an **ambient, per-turn Chat Context** (the open work item, and the open Loop Editor's loop) attached to each message, rather than by binding context onto the session. Two parts of this are non-obvious and were the result of real trade-offs.

**The chat keeps its scratch directory as cwd for the session's whole life; the open work item's worktree is reached by absolute path, not by becoming the cwd.** A weaker model is less confused when its cwd *is* the repo worktree, so the obvious design is to run the chat in that worktree — but Claude Code keys its resumable session JSONL to the *encoded cwd* (`ClaudeCodeAdapter.cs`), so switching cwd mid-session breaks session resume and the chat loses multi-turn memory. We rejected "roam the cwd per turn" (and the per-working-directory session map it would have required) in favour of a stable scratch cwd plus a per-turn `--add-dir <worktreePath>` for the open work item's **active (non-terminal) run** only. Worktree access is therefore gated by both the session's `read`/`write`/`execute` grant and the presence of an active run.

**Loop-editor edits made by the AI are applied to transient client state and are never persisted by the AI — only the human can save.** The loop being edited is client-side React Flow state that may have diverged from the persisted `LoopTemplateVersion`, so the AI reads it from a per-session scratchpad (refreshed each message) and writes a full `ild-loop-template/v1` document back over `/hubs/chat`, which the editor validates with the existing `parseImportFile` and direct-applies to the canvas. The no-save guarantee is **structural**: the agent is given no persist tool, so the only write to a `LoopTemplateVersion` remains the editor's human-only Save. We rejected a staged accept/reject review layer (contradicts the "UI updates immediately" intent) and patch-style edits (error-prone for a model against node/edge ids).

## Consequences

- **Worktree access is path-based, active-run-only.** Finished-run worktrees (kept on disk per [ADR-0008](./0008-worktree-and-branch-per-run.md)) are not exposed to the chat; a work item with no active run gives the agent scratch only. The `--add-dir` grant is Claude-specific and needs an opencode/pi equivalent to keep adapter parity ([ADR-0009](./0009-adapter-feature-parity.md)).
- **Heavy context is pulled, not pushed.** Only a thin pointer (open work-item id, "loop editor open" flag) enters the model context each turn; the full work item, preview state, and loop JSON are fetched by tools on demand, so token cost is paid only when relevant.
- **Loop edits are fire-and-forget.** The agent gets no structured ack; a rejected (invalid) document leaves the canvas untouched and the agent discovers this by re-reading the loop on its next turn.
- **New ILD-control tools fold under the existing `ild` grant** (work items, preview controls, loop read/edit), rather than adding separate toggles. Run-lifecycle mutation (start/halt/steer) is deliberately excluded from this surface.
